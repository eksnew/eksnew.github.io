1. Definition

Dynamic Programming is an algorithmic technique with the following properties.

  • It is mainly an optimization over plain recursion. Wherever we see a recursive solution that has repeated calls for the same inputs, we can optimize it using Dynamic Programming.

  • The idea is to simply store the results of subproblems so that we do not have to re-compute them when needed later. This simple optimization typically reduces time complexities from exponential to polynomial.

  1. Recurrence Relation:

    • To reach step n, you can either:
      • Take one step from n-1 to n, or
      • Take two steps from n-2 to n.
    • Therefore, the total number of ways to reach step n is the sum of the ways to reach n-1 and n-2. This can be expressed mathematically as:
    • This recurrence relation is identical to the Fibonacci sequence, where each term is the sum of the previous two terms.

  2. Memoization:

    • By storing the result of self._climb(n - 1) + self._climb(n - 2) in self.memo[n], we save it in our dictionary to avoid recalculating it in future recursive calls.
    • This optimizes the solution by reducing the time complexity, as each distinct n value is computed only once. Without this, we would have overlapping recursive calls, leading to a time complexity of (O(2^n)).

  3. Recursive Breakdown:

    • When _climb(n) is called, it breaks down into _climb(n-1) and _climb(n-2), which in turn break down further until they reach the base cases (n == 1 or n == 2).
    • By summing these, we effectively build up from smaller subproblems to solve the larger problem.

For example, if n = 5:

  • ways(5) = ways(4) + ways(3)

  • To find ways(4), it recursively calls ways(3) + ways(2), and so on, until it reaches the base cases.

  • Each result is stored in self.memo, allowing the function to quickly look up previously computed values.

We aim to determine the minimum number of coins required to make a given amount.
To achieve this, we maintain a memory array where each cell represents the minimum coins needed to reach that specific amount.
The logic starts by initializing the array:

  • For amount = 0, the minimum coins required is 0.

  • For all other amounts, the initial value is set to infinity (inf) since we haven’t computed the required coins yet.
    To simplify calculations, the array has amount + 1 cells, indexed from 0 to amount.

Suppose we have coins [1, 2, 5] and aim to compute the minimum coins needed to reach amount = 6. Here’s how the process unfolds:

We iterate through each index i (representing an amount) and compute the minimum coins required for that amount by considering all possible coins c where c ≤ i.
For each coin, we calculate the cost as:

Step 1: i = 1

  • Only c = 1 can contribute.

  • Using it, the value becomes 1 + coins_needed[0] = 1.

Step 2: i = 2

  • Two coins (1 and 2) can contribute:

    • Using c = 1: 1 + coins_needed[1] = 2
    • Using c = 2: 1 + coins_needed[0] = 1 (smaller)

Step 3: i = 3

  • Coins 1 and 2 are considered:

    • Using c = 1: 1 + coins_needed[2] = 2
    • Using c = 2: 1 + coins_needed[1] = 2
      Both options yield the same result.

Step 4: i = 4

  • Coins 1 and 2 are valid:

    • Using c = 1: 1 + coins_needed[3] = 3
    • Using c = 2: 1 + coins_needed[2] = 2 (smaller)

Step 5: i = 5

  • Coins 1, 2, and 5 are valid:

    • Using c = 1: 1 + coins_needed[4] = 3
    • Using c = 2: 1 + coins_needed[3] = 3
    • Using c = 5: 1 + coins_needed[0] = 1 (smallest)

Step 6: i = 6

  • Coins 1, 2, and 5 are valid:

    • Using c = 1: 1 + coins_needed[5] = 2
    • Using c = 2: 1 + coins_needed[4] = 3
    • Using c = 5: 1 + coins_needed[1] = 2 (equal to the first option)

4. Prob. 300. Longest Increasing Subsequence

Given an integer array nums, return the length of the longest strictly increasing subsequence.

4.1. Sol. 1: Brute Force

4.1.1. Implementation

4.1.2. Analysis

The brute force approach explores all possible subsequences by recursively deciding for each element whether to include it in the subsequence or skip it.

  • Time Complexity: O(2^n).
    Each element has two choices (include or skip), leading to an exponential growth in calls.

  • Space Complexity: O(n).
    Due to the recursion stack.

This approach performs many redundant calculations, making it inefficient for large input sizes.

Improvement: the state of the solution can be represented as dp[i][prev].
i.e., the length of LIS statring at i, last pick element is prev.
